1. Visão Geral do Produto
CloudNotes é um aplicativo de notas minimalista, multi-usuário e auto-hospedado, inspirado no
Simplenote. O objetivo é oferecer uma experiência de escrita rápida, limpa e confiável, com
sincronização em tempo real entre dispositivos, armazenamento em PostgreSQL e integração
com MinIO para backups e anexos.
O sistema é composto por três camadas:
•
•
•

API REST (Node.js + Fastify) — lógica de negócio, autenticação e persistência
Frontend Web / PWA (React) — interface responsiva, instalável como app
Infraestrutura (PostgreSQL + MinIO) — já provisionada na VPS do cliente

4. Especificação da API REST
Base URL: https://<seu-dominio>/api/v1
Todas as rotas (exceto autenticação) requerem o header:
Authorization: Bearer <access_token>

Formato de resposta padrão:
{ "data": <payload>, "meta": { "timestamp": "...", "version": "1" } }

Formato de erro padrão:
{ "error": { "code": "NOTE_NOT_FOUND", "message": "Nota não encontrada",
"statusCode": 404 } }

6. Segurança
•
•
•
•
•
•
•
•

Senhas: bcrypt com salt rounds = 12
JWT: access token (15min) + refresh token (30 dias, HttpOnly cookie, SameSite=Strict)
Rate limiting: 10 req/s por IP nas rotas de auth; 100 req/s por usuário nas demais
CORS: whitelist de origens configurável via variável de ambiente
Helmet.js: headers de segurança padrão (CSP, HSTS, X-Frame-Options)
Input sanitization: strip de HTML em todos os campos de texto para prevenir XSS
Isolamento por usuário: todas as queries incluem user_id para evitar acesso cross-user
MinIO: URLs pré-assinadas com expiração de 1 hora; bucket não é público

9.2 Autenticação com Refresh Token
7. Login → API retorna accessToken (body) + refreshToken (cookie HttpOnly)
8. Frontend armazena accessToken em memória (não em localStorage)
9. Axios interceptor adiciona Bearer token em todas as requisições
10. Ao receber 401 → interceptor chama POST /auth/refresh automaticamente
11. Se refresh válido → novo accessToken → repete a requisição original
12. Se refresh expirado → redireciona para /login

9.3 Sincronização Offline
13. Usuário fica offline → Service Worker detecta
14. Edições são salvas no IndexedDB com flag pendingSync: true
15. Indicador 'Sem conexão — mudanças serão sincronizadas' aparece na UI
16. Conexão restaurada → Service Worker emite evento 'online'
17. Frontend percorre fila de pendingSync e envia as edições em ordem
18. Conflitos (nota editada em outro dispositivo) → versão do servidor prevalece, versão
local é salva como rascunho
